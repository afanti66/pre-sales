# 总体方案章节

## 一、业务架构设计

### 1.1 组织角色

| 角色 | 职责范围 | 系统权限 | 设置人数 |
|------|---------|---------|:--------:|
| 系统管理员 | 系统配置、应用接入、全局策略 | 全功能管理权限 | 2-3人 |
| 安全审计员 | 审计日志查看、告警配置、安全报表 | 审计模块只读+告警配置 | 2-5人 |
| 组织管理员 | 组织架构维护、角色分配 | 组织管理+角色管理 | 每部门1人 |
| 普通用户 | 个人信息管理、密码重置、应用访问 | 自助服务+SSO登录 | 全员 |
| 应用管理员 | 应用注册、密钥管理、接入配置 | 应用管理模块 | 每应用1人 |

**权限层级设计：**
```
系统级管理员（全局配置）
  └── 安全审计员（只读+告警）
        └── 组织管理员（本组织范围）
              └── 普通用户（仅个人数据）
```

### 1.2 业务对象

| 业务对象 | 属性 | 行为 | 关系 |
|---------|------|------|------|
| 组织(Organization) | 编码、名称、层级、路径、负责人 | 创建/变更/合并/撤销 | 树形层级，父-子关系 |
| 用户(User) | 账号、密码、姓名、手机、邮箱 | 注册/审核/激活/禁用/删除 | 从属组织，持有角色 |
| 角色(Role) | 编码、名称、类型、权限集 | 创建/分配/继承/互斥 | 用户可持有多个 |
| 权限(Permission) | 资源ID、操作类型、约束条件 | 授权/回收/判定 | 从属于角色 |
| 应用(Application) | 应用ID、名称、协议类型 | 注册/接入/轮换 | 使用SSO服务 |
| 审计日志(AuditLog) | 时间戳、操作人、类型、结果 | 记录/检索/哈希验证 | 关联用户+操作 |

### 1.3 业务规则

| 编号 | 规则名称 | 规则描述 |
|:----:|---------|---------|
| BR-001 | 角色互斥 | 同一用户不可同时分配审计员和系统管理员角色 |
| BR-002 | 密码策略 | 密码长度≥8位+大小写+数字+特殊字符，90天更换 |
| BR-003 | 登录锁定 | 连续5次失败，锁定30分钟 |
| BR-004 | 最小权限 | 用户只能拥有完成工作所需的最小权限集 |
| BR-005 | 会话超时 | 空闲30分钟登出，绝对超时12小时 |
| BR-006 | 数据隔离 | 组织管理员只能管理本组织及下属组织用户 |

### 1.4 业务流程

**1.4.1 用户入职流程**
```
HR推送新员工信息 → 系统自动创建用户(待激活)
  → 发送入职通知（含密码+登录地址）
  → 员工首次登录 → 强制改密 → 分配角色
  → 用户可访问已授权应用
```

**1.4.2 用户离职流程**
```
HR推送离职通知 → 系统自动禁用账号
  → 终止所有活跃会话 → 清除缓存
  → 通知各业务系统注销Session
  → 90天后逻辑删除
```

**1.4.3 权限变更流程**
```
管理员发起变更 → 系统校验互斥规则
  → 提交审批 → 审批通过→更新角色映射
  → 清除权限缓存 → 写入审计日志
```

### 2.3 工作原理

**2.3.1 微服务通信原理**
各分系统间通过REST API同步通信，通过RabbitMQ异步解耦。API网关作为统一入口，负责认证拦截和请求路由。服务发现采用Nacos，配置管理采用Nacos Config。

**2.3.2 认证工作原理**
```
用户提交凭证 → 认证服务验证 → 验证通过 → 生成JWT
  → JWT含用户ID/角色/有效期 → 返回客户端
  → 客户端携带JWT访问资源 → API网关校验签名
  → 通过→路由到目标服务 → 服务解析JWT获取用户信息
```

**2.3.3 Token工作原理**
- AccessToken：JWT格式，RS256签名，有效期15分钟
- RefreshToken：不透明令牌，Redis存储（key: refresh:{userId}:{uuid}），有效期7天
- AccessToken过期→RefreshToken换取新AccessToken
- RefreshToken过期→重新登录

**2.3.4 权限判定工作原理**
```
请求到达 → 权限拦截器 → 从JWT提取角色列表
  → 查询Redis缓存(角色→权限映射)
  → 缓存命中→直接判定
  → 缓存未命中→查询数据库→写入Redis
  → 比对请求资源+操作是否在权限集中
  → 通过/拒绝 → 写入审计日志
```

### 2.4 工作流程

**2.4.1 用户登录认证工作流**
```
[客户端] → [API网关] → [认证服务] → [Redis/DB]
  ① 登录请求
  ② 路由到认证服务
  ③ 查询用户信息、bcrypt比对
  ④ 生成JWT + RefreshToken
  ⑤ 返回Token → 客户端存储
```

**2.4.2 SSO跨应用访问工作流**
```
用户访问应用A → 重定向到认证中心
  → 输入凭证 → 认证中心验证 → 生成全局Session
  → 回调应用A → 应用A获取Token
  → 用户访问应用B → 检测到全局Session
  → 自动生成授权码 → 回调应用B → 自动登录
```
